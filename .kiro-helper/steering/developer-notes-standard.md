# Developer Notes Standard

When generating developer notes for a completed module (feature), always:

1. Create a folder at `developer-notes/{module-name}/` with multiple markdown files
2. Write in plain English for a non-technical manager to review
3. Be conversational and audio-friendly — write as if speaking to the manager directly
4. Explain WHAT was built, WHY it was built that way, and HOW to verify it's correct
5. No shortcuts — cover every task group with full explanations
6. Include a main index file and separate files for each major area (domain, backend logic, API, frontend, testing)
7. Reference specific requirements so the manager can cross-check against the spec
8. Explain business rules in plain language, not code
9. Include a "Questions to Ask the Developer" section in each file
10. End with a summary checklist the manager can use for sign-off
